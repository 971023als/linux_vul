# 기여 가이드

이 저장소는 **방어형 보안 연구 문서 저장소**입니다.  
기여는 환영하지만, 이 저장소의 목적과 운영 원칙을 반드시 숙지한 뒤 참여해 주세요.

---

## 받는 기여

다음 유형의 기여를 환영합니다:

- **문서 개선**: README, docs/ 폴더의 오탈자 수정, 표현 개선, 구조 정리
- **방어 관점 보강**: 방어팀, DevSecOps, IAM, AI 거버넌스 관점의 설명 추가
- **인사이트 보강**: 공개된 연구 자료나 사실에 기반한 인사이트 추가
- **FAQ 확장**: 실제 질문에 근거한 새로운 질문/답변 쌍 추가
- **아키텍처 설명 개선**: 개념적 계층 구조 설명의 명확성 향상
- **접근성 개선**: UI 컴포넌트의 접근성(WCAG) 향상
- **번역 기여**: 영어 버전 문서 번역 (계획 중)
- **데이터 파일 개선**: `data/` 폴더의 타입 안전성 또는 내용 정확성 개선
- **테스트 추가**: 기존 데이터/컴포넌트에 대한 단위 테스트 추가

---

## 받지 않는 기여

아래 유형은 어떤 이유에서도 수용하지 않습니다:

- **공격 코드**: 익스플로잇, RCE 재현 코드, 페이로드, 스캐너 로직
- **취약점 재현 절차**: 실제 공격 단계, 우회 방법, 침투 자동화 로직
- **자격증명 관련 코드**: 패스워드 탈취, 세션 하이재킹, 인증 우회
- **실존 시스템 점검 결과**: 허가받지 않은 시스템의 실제 취약점 데이터
- **과장된 주장**: 검증되지 않은 기술적 사실, fabricated detail
- **비인가 도구 연동**: 실제 공격 도구와 연동되는 코드나 설정

## PR 작성 원칙

1. **변경 목적을 명확히**: PR 제목과 본문에 무엇을, 왜 변경했는지 기술하세요.
2. **사실 근거 포함**: 새 내용을 추가할 경우, 출처 또는 근거 자료를 함께 제시하세요.
3. **작은 단위로**: 하나의 PR은 하나의 목적에 집중하세요.
4. **공격 재현 가능성 검토**: PR을 올리기 전 "이 내용이 공격 실행에 활용될 수 있는가"를 스스로 검토하세요. 의심스러우면 Issue를 먼저 열어 논의하세요.
5. **한국어 우선**: 문서는 한국어 중심으로 작성하되, 기술 용어는 영어 혼용 가능합니다.

---

## 사실 검증 원칙

- 기술적 사실을 추가할 때는 공개 자료(논문, 공식 문서, CVE, 보안 블로그)를 참조로 제시하세요.
- 참조 불가능한 내용은 "개념적 설명" 또는 "가정 시나리오"로 명시하세요.
- Madhat Labs 연구 환경에 대한 실제 시스템 정보(IP, 자격증명, 내부 구성)는 어떠한 경우에도 포함하지 마세요.

---

## 위험한 재현 가능성 억제 원칙

이 저장소는 공격자에게 "따라 할 수 있는 단계"를 제공하지 않습니다.  
기여 시 다음 질문을 스스로 해보세요:

> "이 내용을 읽은 사람이 비인가 시스템에 피해를 줄 수 있는 행동을 할 수 있는가?"

"예"라면 기여하지 마세요.  
확신이 없으면 Issue에서 먼저 논의하세요.

---

## 기여 절차

1. 이 저장소를 Fork합니다.
2. 기능/수정용 브랜치를 생성합니다: `git checkout -b docs/improve-faq`
3. 변경 후 커밋합니다: `git commit -m "docs: FAQ 항목 3개 추가"`
4. Fork 저장소에 Push합니다: `git push origin docs/improve-faq`
5. Pull Request를 생성합니다.

## 질문과 논의

기여 전에 아이디어를 논의하고 싶다면 GitHub Issue를 열어 주세요.  
Issue 템플릿 중 **문서 요청** 또는 **방어 논의** 템플릿을 활용하세요.
