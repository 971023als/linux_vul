# 전자금융 웹·모바일·HTS 보안진단 기준 저장소

[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](LICENSE.md)

## 한 줄 소개

전자금융업 웹, 모바일 앱, HTS 환경의 보안 점검항목을 위험도, 법규 근거, 감독규정, 시행세칙 기준으로 정리하는 **방어형 보안진단 문서 저장소**입니다.

---

## 안전 고지

이 저장소는 **방어·문서화·교육 목적**의 보안진단 기준 정리 저장소입니다.

- 점검항목은 통제 검증, 위험 식별, 보안성 검토, 증적 확보 관점으로만 기술합니다.
- 공격 재현 절차, exploit 코드, 우회 방법, 자동 공격 스캐너 로직은 포함하지 않습니다.
- 실제 운영 시스템에 대한 무단 점검에 활용할 수 없습니다.
- 모든 진단 활동은 명시적 허가를 받은 환경에서만 수행해야 합니다.

---

## 프로젝트 개요

전자금융업자는 「전자금융거래법」 제21조에 따라 정보시스템의 안전성 확보 의무를 집니다. 이 저장소는 금융감독원이 정의한 보안진단 기준을 바탕으로 웹, 모바일 앱, HTS 환경에서의 66개 점검항목을 체계적으로 정리합니다.

| 항목 | 내용 |
|------|------|
| 점검 대상 | 전자금융업 웹, 모바일 앱, HTS |
| 점검항목 수 | 총 66개 |
| 위험도 분류 | 1 (최고) ~ 5 (최저) |
| 주요 법규 근거 | 전자금융거래법 제21조, 전자금융감독규정 제15조·제16조, 시행세칙 별표 3 |
| 저장소 성격 | 방어형 문서 저장소 (공격 코드 미포함) |

## 기준표 구조

점검항목 기준표는 다음 필드로 구성됩니다.

| 필드 | 설명 |
|------|------|
| 번호 | 항목 고유 번호 (1~66) |
| 점검항목명 | 항목 명칭 |
| 점검 영역 | 웹 / 모바일 / HTS / 공통 |
| 위험도 | 1 (최고위험) ~ 5 (참고) |
| 통제 목적 | 해당 항목이 방어하는 보안 목표 |
| 법규 근거 | 전자금융거래법·감독규정 조항 매핑 |
| 증적 유형 | 점검 시 수집·확인해야 할 증적 유형 |
| 비고 | 예외 조건, 추가 확인사항 |

> 상세 66개 항목 전체 목록은 향후 `data/inspection-items.ts` 또는 `docs/inspection-items.md`로 분리하여 관리할 예정입니다.

## 주요 점검 영역

### 1. 전자금융 거래 보호

전자금융거래의 무결성, 부인방지, 이중 거래 방지를 위한 통제 검증 항목입니다.

- 거래 무결성 검증 (요청-응답 간 금액·수신자 위변조 여부)
- 거래 재전송(Replay) 방지 메커니즘 적용 여부
- 전자서명 및 MAC 값 검증 절차
- 이상거래탐지시스템(FDS) 연동 여부

### 2. 인증·인가·세션 통제

사용자 신원 확인과 권한 통제, 세션 수명주기 관리 항목입니다.

- 로그인 실패 횟수 제한 및 잠금 정책
- 다중 인증(MFA) 적용 여부 (이체, 비밀번호 변경 등 고위험 거래)
- 세션 타임아웃 및 만료 후 무효화 처리
- 수평·수직 접근통제 검증 (타인 계좌·화면 접근 여부)
- JWT·토큰 만료, 서명 검증, 재사용 방지

### 3. 모바일·HTS 클라이언트 보안

모바일 앱 및 HTS 클라이언트 환경의 단말 보안 통제 항목입니다.

- 루팅·탈옥 단말 탐지 및 실행 제한
- 앱 무결성 검증 (코드 서명, 변조 탐지)
- 중요 데이터 로컬 저장 여부 (SharedPreferences, SQLite, 로그)
- 화면 캡처·스크린샷 방지 설정
- 디버깅 모드 제한 (안드로이드 debuggable flag, iOS 등)
- HTS 클라이언트 실행 파일 무결성 검증

### 4. 중요정보 보호

민감 정보의 전송·저장·노출 방지 통제 항목입니다.

- 계좌번호, 주민번호, 카드번호 마스킹 처리
- 비밀번호 해시 저장 여부 (bcrypt, argon2 등 강화 해시)
- 소스코드 내 하드코딩된 자격증명·API 키 여부
- 응답 헤더·에러 메시지 내 스택 트레이스·버전 정보 노출 여부
- 개인정보 전송 시 암호화 적용 여부

### 5. 웹 애플리케이션 보안

입력값 검증, 출력 처리, 웹 취약점 통제 항목입니다.

- 입력값 검증 항목 (SQL Injection, SSTI, XXE, Command Injection 등 점검항목 기준)
- XSS 방지 출력 이스케이프 적용 여부
- CSRF 토큰 적용 여부 (상태 변경 요청)
- 파일 업로드 제한 (확장자, 크기, 악성코드 검사)
- SSRF 방지를 위한 서버 측 요청 출처 검증
- Open Redirect 취약점 통제 여부

> ※ 위 항목은 점검 명칭으로만 언급합니다. 재현 방법 및 악용 절차는 이 저장소에 포함하지 않습니다.

### 6. 서버·통신·운영정보 보호

서버 설정, 통신 암호화, 운영 정보 노출 방지 항목입니다.

- HTTPS 강제 적용 (HSTS 헤더 포함)
- TLS 버전 및 암호화 스위트 적절성 (TLS 1.2 이상, 취약 알고리즘 제외)
- 보안 헤더 설정 (CSP, X-Frame-Options, X-Content-Type-Options 등)
- 관리자 페이지 외부 노출 여부
- 디렉터리 리스팅 차단 여부
- 에러 페이지 운영 정보(스택 트레이스, 버전) 노출 방지

## 법규 및 감독기준 매핑

| 법규·기준 | 조항 | 관련 점검 영역 |
|-----------|------|----------------|
| 전자금융거래법 | 제21조 (안전성 확보의무) | 전 영역 공통 |
| 전자금융감독규정 | 제15조 (정보시스템 보안대책) | 서버·통신·운영정보 보호 |
| 전자금융감독규정 | 제16조 (전자금융거래의 안전성 확보) | 인증·인가·거래 보호 |
| 시행세칙 별표 3 | 정보시스템 보안대책 | 웹·모바일·HTS 전 영역 |

> 각 점검항목별 세부 조항 매핑은 `docs/legal-mapping.md`에서 관리할 예정입니다.

## 활용 시나리오

**보안진단 준비**: 자체 점검 전 체크리스트로 활용하여 점검 범위와 증적 목록을 사전에 정의합니다.

**위험도 우선순위 수립**: 위험도 1~2 항목을 우선 점검하여 고위험 취약점을 조기에 식별합니다.

**감독기관 대응**: 금융감독원 현장검사 및 자체 보안점검 결과 보고서 작성 시 항목별 법규 근거 확인에 활용합니다.

**개발팀 교육**: 보안 요구사항을 개발 단계에서 이해하고 내재화하는 교육 자료로 활용합니다.

## 비목표

이 저장소는 다음을 제공하지 않습니다.

- 취약점 자동 탐지 스캐너 또는 공격 도구
- 특정 시스템에 대한 실제 취약점 재현 코드
- 우회 기법, exploit 페이로드, RCE 실습 코드
- 특정 금융기관 또는 시스템에 대한 점검 결과
- 법적 컴플라이언스 인증 또는 보장

## 보고서화 방향

점검 결과는 다음 구조로 보고서화할 수 있습니다.

```
[점검항목 번호] 항목명
├── 위험도: 1~5
├── 법규 근거: 전자금융거래법 제21조 / 감독규정 제15조 등
├── 통제 목적: (해당 통제가 방어하는 보안 목표)
├── 점검 결과: 양호 / 취약 / 해당없음
├── 증적: (캡처, 로그, 설정값 등 수집 내용)
└── 조치 권고: (취약한 경우 개선 방향 — 개념 수준)
```

상세 보고서 템플릿은 `docs/report-template.md`로 분리하여 관리할 예정입니다.

## 면책 조항

이 저장소의 내용은 공개된 법규·감독규정·연구 자료를 기반으로 작성된 참고 자료입니다.

- 법적 컴플라이언스 달성을 보장하지 않습니다.
- 특정 금융기관의 보안 수준을 평가하거나 인증하지 않습니다.
- 보안 결정 전에 전문가 자문을 구하시기 바랍니다.
- 이 내용의 오용으로 인한 결과에 대해 저장소 관리자는 책임을 지지 않습니다.
