# 보안 아키텍처 개요

이 문서는 Madhat Labs 연구에서 다루는 웹·클라우드·IAM·에이전트 계층의
개념적 보안 아키텍처를 설명합니다.

---

## 계층 구조 개요

```
┌─────────────────────────────────────┐
│           웹 애플리케이션 계층         │
│  입력 검증 · 인증 · 세션 · CSP        │
├─────────────────────────────────────┤
│           AI 에이전트 계층            │
│  신뢰 경계 · 툴 호출 · 샌드박싱        │
├─────────────────────────────────────┤
│           클라우드 인프라 계층         │
│  IAM · VPC · 시크릿 관리 · 공급망     │
├─────────────────────────────────────┤
│        Identity & Access Management │
│  RBAC · PoLP · Conditions · 감사     │
└─────────────────────────────────────┘
```

※ 이 다이어그램은 개념적 설명을 위한 것이며, 실제 구현 세부사항을 반영하지 않습니다.

---

## 웹 애플리케이션 계층

### 주요 보안 통제

- **입력 검증**: 모든 외부 입력에 대해 타입, 길이, 형식 검증 수행
- **출력 이스케이프**: 사용자에게 노출되는 데이터의 XSS 방지 처리
- **인증·인가**: JWT, 세션 기반 인증과 역할 기반 접근 제어
- **보안 헤더**: CSP, HSTS, X-Frame-Options, X-Content-Type-Options

### AI 에이전트 통합 시 추가 고려사항

- 사용자 입력이 에이전트 컨텍스트로 전달되기 전 신뢰 레이블 부여
- 에이전트 응답을 사용자에게 노출하기 전 출력 검증 수행

---

## AI 에이전트 계층

### 신뢰 경계 설계 원칙

에이전트가 처리하는 입력은 신뢰 수준에 따라 분류되어야 합니다:

1. **신뢰 높음**: 시스템 프롬프트, 개발자 정의 지시
2. **신뢰 중간**: 검증된 사용자 입력
3. **신뢰 낮음**: 외부 데이터(웹, 파일, DB 결과), 외부 에이전트 응답

### 툴 호출 보안

- 툴 목록을 명시적 화이트리스트로 관리
- 툴 입력 파라미터 검증 (타입, 범위, 허용값)
- 툴 호출 깊이 및 횟수 제한
- 실행 환경 샌드박싱

## 클라우드 인프라 계층

### IAM 설계 원칙

- 에이전트용 서비스 계정은 최소 권한으로 설계
- `*` 와일드카드 리소스 지정 지양
- Conditions로 시간, IP, MFA 조건 추가
- 역할 체인(role chaining) 검증 및 제한

### 네트워크 분리

- 에이전트 실행 환경을 별도 VPC 또는 서브넷으로 분리
- 에이전트에서의 외부 인터넷 접근을 화이트리스트 도메인으로 제한
- Security Group을 통한 포트 수준 접근 제어

---

## 멀티 에이전트 신뢰 모델

멀티 에이전트 아키텍처에서 신뢰는 에이전트 간에 자동으로 전파되어서는 안 됩니다.

### 권장 패턴

```
Orchestrator
  ├─ Sub-agent A 호출 → 응답 수신 → 스키마 검증 후 소비
  └─ Sub-agent B 호출 → 응답 수신 → 스키마 검증 후 소비
```

오케스트레이터는 하위 에이전트의 응답을 실행 지시로 처리하지 않고,
구조화된 데이터로 파싱한 후 사용해야 합니다.
